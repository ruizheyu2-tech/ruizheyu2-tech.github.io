## 代码文件分别是干什么用的

1. 启动与配置层

	1. `multicore_ext.sh` 是“包工头”。它根据 CPU 核心数，并行启动多个 `root2root` 进程，分配任务范围。
    
	2. `setup.h` 是“法律/配置单”。它定义了全局物理常数（如 `SL_RS`）、阈值、时间窗（`COIN_WINDOW`）和数据结构（如 `FIT_RESULT`）。
    
	3. `root2root.cpp` 是“车间主任”。它是主程序入口，负责初始化共享内存环境，并启动 `SortEvent` 循环。
    
2. 解码与生产层 (Producer)

	1. `CAEN_DGTZ.cpp` (+.h) 是“翻译官”。它读入磁盘上的 `DG_TREE` (生数据)，通过位操作将其翻译成 `Waveform_t` (波形) 和基础信息。
    
	2. `WaveProcess.cpp` (+.h) 是“精算师”。它接收波形，利用梯形滤波(PHA)、微分(RCCR2)或恒比定时(CFD)算法，计算出精确的 `Energy` 和 `Time`。
    
	3. `SortEvent.cpp` (+.h) 是“调度员”。它指挥“翻译官”解码，指挥“精算师”算波形，处理堆积逻辑，最后把完整的 `DPP_PHA_Event_t` 塞入共享内存。
    
3. 传输与消费层 (Consumer)

	1. `DataShm.cpp` (+.h) 是“传送带”。它管理环形缓冲区和信号量，确保数据安全、高速地从解码线程流向物理分析线程。
    
	2. `mapping.cpp` (+.h) 是“物理学家”。它从“传送带”取下数据，进行复杂的物理逻辑判断（衰变链搜索、反符合、死时间恢复），最终生成物理结果文件 `_map.root`
## `_map.root`和`_chain.root`和`_tmp.root`文件有什么差异

这三个文件代表了数据处理流程中**三个完全不同的阶段**。

从数据流向来看，它们的演变关系是： **原始二进制数据 → `_tmp.root` (清洗/重算) → `_map.root` (物理构建) → `_chain.root` (历史关联)**

### 1. `_tmp.root`：清洗后的硬件数据

**本质**：这是“硬件视角”的数据。它是原始数据的直接翻译和波形重算结果，但数据依然是按“板卡-通道”组织的，没有物理概念。

- **差异特征**：
    
    - **组织方式**：按板卡分 Tree（如 `DGTZ00`, `DGTZ01`）。
        
    - **包含内容**：每个 Entry 是一个独立的触发信号（Signal）。
        
    - **物理意义**：这里只有“第0号板子第1通道收到了一个信号”，不知道这个信号是 DSSD 还是 Ge。
        
    - **是否包含波形**：通常不包含（为了省空间），但包含了由波形算出来的精确能量和时间。

**关键代码**：SortEvent.cpp
	
 **创建文件**
```cpp
sprintf(fileout,"%s%s%05d_tmp.root",...);
TFile *FileOut = new TFile(fileout,"RECREATE");
```
	
 **Tree定义**
```cpp
// 按板卡数量循环创建 Tree
for(int b=0; b<MAXNB; b++) {
    sprintf(TreeName,"DGTZ%02d",b); // DGTZ00, DGTZ01...
    DGTZ[b] = new TTree(TreeName,...);
    // ... Branch 定义 ...
}
```
	
 **波形重算 (可选)**： 如果有 `DECODE_WAVE_PHA` 宏，代码会调用 `WaveProcess` 里的 `Get_Trace_Trigger_RCCR2` 和 `GetPHAResult` 重新计算 `Energy` 和 `TriggerTimeTag`，覆盖掉硬件原始值。
	
 **填充**
```cpp
DGTZ[DPPPHAEvent_Data.BdNo]->Fill(); // 填入对应的板卡 Tree
```

### 2. `_map.root`：构建好的物理事件

把硬件地址映射成了物理探测器，并把同一时刻发生的所有信号组合成了一个**物理事件**。

- **差异特征**：
    
    - **组织方式**：通常只有一个主 Tree（如 `Tree_Map`）。
        
    - **包含内容**：每个 Entry 是一个**符合事件**。
        
        - 例如：一个 Entry 可能包含“DSSD正面能量 + DSSD背面能量 + Ge探测器能量 + MWPC时间”。
        
    - **处理逻辑**：
        
        - **映射 (Mapping)**：把 `Bd=0, Ch=1` 翻译成 `DSSD_Front_Strip_10`。
            
        - **符合 (Coincidence)**：利用 `COIN_WINDOW`，把时间相近的信号打包。
            
        - **加和 (Addback)**：把相邻条的能量加起来（处理电荷分际）。
    
**关键代码**：`mapping.cpp`中的`Mapping_Data()` (线程入口) -> `Sort_DGTZ_DATA()` -> `Analyze_Coin_Event()`

**读取共享内存**： 从 Shm 读取 `DPP_PHA_Event_t` 数据。

**时间窗符合逻辑**：
```cpp
// 在 mapping.cpp 中通常会有类似这样的逻辑（虽然你没贴出具体的时间窗循环，但这是 Analyze_Coin_Event 的核心）
if (current_time - first_time < COIN_WINDOW) {
    // 这是一个事件里的，打包！
} else {
    // 时间窗过了，把上一个事件填入 Tree，开始新事件
    Tree_Map->Fill();
}
```
**探测器处理**： `Process_DSSD_MulSignal`：处理 DSSD 的多重信号。 `Process_Clover_Signal`：处理 Ge 探测器的信号。
### 3. `_chain.root`：衰变链历史记录

不再关注瞬间的符合，而是关注**同一个位置在长达几秒甚至几分钟内的历史记录**。

- **差异特征**：
    
    - **组织方式**：Tree 结构通常包含 `Implant`（注入）, `Alpha1`, `Alpha2` 等分支。
        
    - **包含内容**：完整的**衰变链**。
        
    - **核心逻辑**：**位置关联 (Position Correlation)**。
        
        - 如果在 DSSD 的像素 (X=10, Y=20) 发生了一个反冲核注入事件。
            
        - 过了 5ms，在同一个像素 (X=10, Y=20) 又发生了一个 Alpha 衰变。
            
        - 代码会把这两个不同时间的事件链接起来，构成一条链。
    
**关键代码**：`mapping.cpp`中的`Fill_Decay_Chain_Data` (或类似的搜索函数)

**像素网格搜索**： 代码会维护一个巨大的二维数组（代表 DSSD 的每个像素），记录该像素上一次发生的事件。
```cpp
// 伪代码逻辑
if (is_alpha) {
    // 去查这个像素 (posx, posy) 之前有没有注入事件
    last_event = Pixel_History[posx][posy];
    if (Check_Correlation(last_event, current_alpha)) {
        // 找到关联！填入 Chain Tree
        Fill_Decay_Chain_Data(...);
    }
}
```
**定义链结构**： 在 `mapping.h` 里定义的 `Chain_Data` 结构体，以及 `mapping.cpp` 里对 `Tree_Chain` 的 `Branch` 定义。
### 总结

| 文件名               | 生成阶段  | 核心文件            | 数据特点                              |
| ----------------- | ----- | --------------- | --------------------------------- |
| **`_tmp.root`**   | 解码与重算 | `SortEvent.cpp` | 按板卡存储，无物理关联，包含重算的精确能量/时间。         |
| **`_map.root`**   | 物理构建  | `mapping.cpp`   | 按事件存储，经过时间符合包含DSSD/Ge /MWPC 等物理量。 |
| **`_chain.root`** | 历史关联  | `mapping.cpp`   | 按衰变链存储，经过位置关联，包含注入-衰变的时间差。        |
## setup.h文件

### 头文件保护与基础依赖
```cpp
#ifndef SETUP_H
#define SETUP_H

#include<stdlib.h>
#include<stdio.h>
#include<stdint.h>
#include"TROOT.h"
```

在 C/C++ 语言中，以井号 `#` 开头的行被称为 **预处理指令。

简单来说，**在编译器真正开始编译代码之前，有一个叫“预处理器”的工具会先扫一遍代码**。这些 `#` 开头的指令就是给这个“预处理器”下达的命令。

1. `#define`：简单的“查找与替换”，意思是**给一个数值或一段代码起个名字**。

	- **语法**：`#define 名字 内容`
    
	- **动作**：预处理器会在整个代码里，把所有的“名字”都替换成“内容”。
    
2. `#ifndef`、`#define`、`#endif`：防止重复包含（头文件卫士），通常出现在头文件的**最开头**和**最结尾**。

	1. **`#ifndef SETUP_H`**：意思是 **"If Not Defined SETUP_H"**（如果 `SETUP_H` 这个记号还没被定义过）。
    
	2. **`#define SETUP_H`**：那么，就定义一下 `SETUP_H`（打个勾，表示我来过这里了），然后继续处理下面的代码。
    
	3. **`#endif`**：结束判断。
	
	- **效果**：
	
		- **第一次**遇到这个文件：`SETUP_H` 没定义 -> 进入内部 -> 定义 `SETUP_H` -> 读取代码。
		
		- **第二次**遇到这个文件：`SETUP_H` 已经被定义了 -> 直接跳到 `#endif` -> **完全忽略中间的内容**。
	
3. `#ifdef`：功能开关，意思是 **"If Defined"**（如果定义了这个名字）。

### 符合时间窗与 DSSD 前后能量一致性判据：
```cpp
//// for search coincidence event 
#define COIN_WINDOW         1500  //unit：ns 符合时间窗
#define COIN_SI_WINDOW      1500  //unit:ns 对于Si的符合时间窗
#define DSSD_Energy_DIFF    200  //in kev, 正背面符合的能量窗区间，大于此区间认为不正常信号，会被丢弃，因此需要根据具体情况设置
```
#### `COIN_WINDOW` 符合时间窗

在核物理实验中，一个物理事件（例如一个重离子打入探测器）往往会同时触发多个探测器。但是，由于以下原因，这些电子学信号到达采集系统的时间并不是严格一致的：
	
- **探测器响应速度不同**：例如，MWPC（气体探测器）的电子漂移速度慢，信号来得晚；而 DSSD（半导体探测器）响应很快。
    
- **线缆长度不同**：信号在同轴电缆中的传输也需要时间。
    
- **电子学处理延迟**：不同的放大器、成形电路延迟不同。
    
只要多个独立的信号到达时间差在这个 1500ns 范围内，程序就认为**它们属于同一个物理事件**（例如：同一个原子核的注入，或者同一次衰变）。

1. 在 `mapping.cpp` 中[[离线数据分析程序详解#在“水位线”之前搜索符合(coincidence)事件：COIN_WINDOW 组包|COIN_WINDOW]]：作为“事件构建器”的核心标准

这是 `COIN_WINDOW` 最核心的用途。在 `Sort_DGTZ_DATA` 函数的 `while` 循环中，程序负责将乱序到达的离散数据包组装成物理事件，意思是把时间差小于COIN_WINDOW的事件都看作同一个事件。

程序选取一个信号的时间戳作为基准 (`Ts_Coin`)，然后向后看。   
```
// mapping.cpp 中的逻辑简化
if(llabs(Ts_Curr - Ts_Coin) < COIN_WINDOW) 
{
    // 这是一个符合事件！
    // 把这个信号加入到当前的 Particle_Data 中
    Get_Particle_Data(&(iter_map->second)); 
}
```

决定了 `Particle_Data` 结构体（也就是一个完整的 Event）里包含哪些信号。例如，它确保了**同一个注入事件**中，`MWPC` 的信号、`DSSD X` 面的信号、`DSSD Y` 面的信号能被打包在一起，从而让后续程序能判断“这是一个有 MWPC 标记的注入事件”。

注意：这里只是在说同一事件，比如捕捉到在DSSD、SSD、Veto上的三个信号是同一个衰变事件引起的，是在说同一个事件的时间窗而不是反应链的时间窗

2. 在 `SortEvent.cpp` 中：辅助处理堆积信号 (Pile-up)

在 `SortEvent.cpp` 的[[离线数据分析程序详解#` ifdef DECODE_WAVE_PHA`：对 pileup 波形做“重新触发 + 重算 PHA + 拆分成多条事件”|DECODE_WAVE_PHA]]部分，当程序通过波形拟合发现一个波形里其实藏着好几个信号（Pile-up）时：
```
// SortEvent.cpp
if(peak_idx > 0 && (Fit_Result.Ts[peak_idx] - Fit_Result.Ts[peak_idx-1])*ts_unit < COIN_WINDOW)
{
    short_pileup_cnt++;
    // ...
    DPPPHAEvent_Data.TriggerTimeTag = ... + short_pileup_cnt*COIN_WINDOW; // 
}
```

这里用 `COIN_WINDOW` 作为一个阈值来判断两个堆积脉冲是“极其紧密”（可能属于同一过程的震荡）还是“稍微分开”（两个快速连续的独立信号）。同时，代码似乎利用 `COIN_WINDOW` 对时间戳做了一个人工偏移，以便后续步骤能识别出这是堆积事件。

我们在看波形有没有叠加信号的时候（希望看到间隔非常短的衰变信号）看的是没有解波形的信号，也就是不用这个PHA和Fit，它俩的作用之一就是给这些信号都拆开

3. 在 `mapping.cpp` 的 `Fill_Decay_Chain_Data` 中：关联 SSD 逃逸粒子

当处理衰变链时，需要处理 Add-back（能量加和）逻辑，特别是针对侧向探测器（SSD）：
```
// mapping.cpp
else if(num > 0 && ptr_chain_data->Delta_Ts[num] < COIN_WINDOW && ...)
```

如果主探测器 (DSSD) 发生衰变后，**1500ns 内** 侧面的 SSD 也有信号，程序会认为这是一个 $\alpha$ 粒子飞出 DSSD 打到了 SSD 上（逃逸事件）。这时程序会将两者关联起来，而不是把 SSD 的信号当成下一代独立的衰变。

#### `COIN_SI_WINDOW` 硅内部符合时间窗

**硅探测器内部（主要是 DSSD 的 X/Y 面、以及同一硅探测器相邻条带）判定“同一次击中/同一个粒子导致的信号”时允许的最大时间差**。  

1. mapping.cpp 中 Analyze_Coin_Event：判定 DSSD 的 X 与 Y 是否“同一次击中”

在 `Analyze_Coin_Event()` 里有核心判断：

```
llabs(Data_DSSDX.Ts[ii] - Data_DSSDY.Ts[jj]) < COIN_SI_WINDOW
```

**只有当 DSSD 前后（X/Y）时间戳差在硅内部符合窗内**，才认为这个 X 条和这个 Y 条可以组成一个像素命中（pos_x, pos_y），继而用于：

- 判定植入(implant)还是衰变(decay)
    
- 做 decay chain 的像素关联（`Data_chain_array[pixel_num]`）

2. mapping.cpp 中 Process_DSSD_MulSignal_new（以及旧版 Process_DSSD_MulSignal）：处理硅条带“相邻多重击中”的加和/合并

如果两个相邻条带在 `COIN_SI_WINDOW` 内都有信号，说明粒子打在缝里了，就认为可能是电荷共享/扩散导致的**一个真实击中分摊到两条**，需要做 add-back：

```
if( llabs(ptr_data->Ts[ii] - curr_Ts) < COIN_SI_WINDOW
    && (ptr_data->Ch[ii] == ptr_data->Ch[ii-1] + 1)
    && ... )
{
    // coin_num++，最后把两条能量合并，并用能量比修正位置（Ch 变成小数）
}
```

程序会将这两个信号合并成一个完整的信号，以保证能量测量的准确性。

#### 影响哪些输出文件

- `COIN_WINDOW` ：贯穿全流程，影响 `tmp`, `map`, `chain` 三个文件。
    
- `COIN_SI_WINDOW` ：主要影响物理分析结果 (`chain` 文件)。

| **变量**             | **物理含义**                           | **在 tmp.root**                 | **在 map.root**                          | **在 chain.root**                       |
| ------------------ | ---------------------------------- | ------------------------------ | --------------------------------------- | -------------------------------------- |
| **COIN_WINDOW**    | **事件总窗口**<br><br>(所有探测器)           | **生效**<br>  <br>(用于堆积信号的时间戳标记) | **核心生效**<br>  <br>(决定哪些信号被打包进同一个 Event) | **生效**<br>  <br>(用于关联 SSD 逃逸粒子)        |
| **COIN_SI_WINDOW** | **硅内部窗口**<br><br>(DSSD X-Y / 相邻条带) | **不生效**                        | **不直接生效**<br><br>(影响后续分析，不影响Raw数据打包)    | **核心生效**<br><br>(决定 X-Y 配对成功率和能量加和准确性) |
#### 怎么找参数以及调整建议

选一轮正常的实验过程中的文件，用root2root得到其map文件，分别画`DSSDX_Ts-MWPC_Ts`，`DSSDX_Ts-DSSDY_Ts`，`DSSDX_Ts-SSD_Ts`，`DSSDX_Ts-Veto_Ts`，`DSSDX_Ts-Ge_Ts`的图。

不管看什么探测器都看只有一个触发的，只有一个触发就只有一个Ts，更清楚。

对于DSSD，正背面能量差不能超限制（如果事件太少可以把能量差的条件去掉，如果依旧少，可以再设置DSSDX_mul<3或 DSSDY_mu<3或同时都设上，前期只是大概给个范围，后面找到真事件后再定准即可）

鉴别出事件后再根据真实事件确定准确时间窗（也要包括上述条件）。

前期对应的命令分别为：

`tr_map->Draw(“DSSDX_Ts[0]-MWPC_Ts[0]”,“DSSDX_mul==1 && DSSDY_mul==1 && MWPC_Ch==0 && MWPC_mul>0 && abs(DSSDX_E[0]-DSSDY_E[0])<200”)`
一般用X面看，也可以用Y面。注意有两个MWPC，要设置看哪一个的Ts，一般用Ch=0的（即产物先经过的那个），还要注意MWPC有信号，不然MWPC_Ts为0，也会减了放在图中，影响看图

注：`xxx_Ts[a]`的a，表示第a+1个触发，不代表Ch，因此若要看`Ch==0`的MWPC，必须限制条件`MWPC_Ch==0`，不能只写`MWPC_Ts[0]`， `MWPC_Ts[0]`表示的是MWPC第一个触发时的Ts，若`Ch==0`的MWPC没触发，`Ch==1`的触发了，则`MWPC_Ts[0]`中填的为`Ch==1`的MWPC的时间。

`tr_map->Draw(“DSSDX_Ts[0]-DSSDY_Ts[0]”,“DSSDX_mul==1 && DSSDY_mul==1 && abs(DSSDX_E[0]-DSSDY_E[0])<200”)`

#### 对比
`COIN_WINDOW`：DSSD与**其它探测器**（DSSD与MWPC，DSSD与SSD，DSSD与Veto及DSSD与Ge）、**DSSD正背面及DSSD相邻条之间**的符合时间窗。
	
 大前提，只要来自同一个事件，触发时间差就必须小于该时间窗，满足该后才会进行下一步更严格的筛选。
	
 根据探测器时间分辨设置，一般为2-3倍FWHM，如果怕漏东西，可以设置大点，实际就是在map中看，把0左右一片包进去就行，再大的范围即使有计数也大概率是偶然的假事件

`COIN_SI_WINDOW`：在满足COIN_WINDOW的前提下对DSSD的正背面以及DSSD相邻条的进一步筛选，这个时间窗在map中不生效，map中的事件都满足COIN_WINDOW，在从map生成chain文件时会代入COIN_SI_WINDOW，也就是对DSSD上的信号进行进一步精确的筛选。
	
 在chain文件中，DSSD与MWPC，DSSD与SSD，DSSD与Veto以及DSSD与Ge的符合时间窗还是COIN_WINDOW，而DSSD正背面以及DSSD相邻条之间的符合时间窗变为COIN_SI_WINDOW，一般设置的COIN_SI_WINDOW≤COIN_WINDOW

`DSSD_Energy_DIFF`：DSSD的X面与Y面符合的能量窗区间，大于此区间认为不正常信号，会被丢弃，需根据具体实验设置，在map中找XY都只触发一次的事件，看DSSDX_E[0]-DSSDY_E[0]即可。由于X面的量程小于Y面，本次实验中X面有大量的超界信号，而这些信号对于Y面来说并未超出量程，因此就可能导致X与Y面能量差别较大，但这些信号又是真实事件不能舍弃，所以<font color=red>本次实验中并未使用该功能</font>
### 不同探测器时间戳修正（把时间对齐到 DSSD）:
```cpp

static const double  GE_TS_CORR[16] = {238.0, 238.0, 238.0, 238.0,       238.0, 238.0, 238.0, 238.0,     -160.0, -160.0, -160.0, -160.0,    0, 0, 0, 0}; //for Ge ts corr, ch start from 288 
//16个Ge通道，前八路走同一套电缆，后八路另一套，后四路没用到就写0
//这次实验只有一个Ge，所以只改第一个数就可以
#define BEAMOn_TS_CORR      70e3
#define BEAMOff_TS_CORR     7e3
#define TARGETOn_TS_CORR      350e3
#define TARGETOff_TS_CORR     650e3
```

1. 修正是什么？ 

	- **核心定义**：时间修正就是“对表”。虽然所有探测器都连在同一个数据获取系统（DAQ）上，但它们记录下来的原始时间戳（Raw Timestamp, `Ts_raw`）是**不齐**的。

	- **理想情况**：一个原子核发生衰变，同时发射出一个 $\alpha$ 粒子（打在 DSSD 上）和一个 $\gamma$ 射线（打在 Ge 上）。理论上，这两个信号的时间戳应该完全相等：$T_{DSSD} = T_{Ge}$。

	- **实际情况**：差了一大截，比如 $T_{Ge}$ 比 $T_{DSSD}$ 晚了 200ns。

	**修正的目标**是通过数学加减法，人为消除这个差异，建立一个**统一的时间参考系**，使得：$T_{Ge\_Corrected} \approx T_{DSSD\_Corrected}$
	
2. 为什么要修正？ 

	1. 信号传输延迟（线缆长度）
	
		- **原理**：电信号在同轴电缆里的传播速度约为 $20cm/ns$（光速的 2/3）。
		
		- **代码对应**：`SSD_TS_CORR` (30ns, 33ns...)
		
		- 如果连接 SSD 的线缆比连接 DSSD 的线缆长 2 米，信号就会晚到 10ns。不同通道的线缆长度很难完全一致，或者经过的前置放大器（Preamp）响应速度不同，导致了这几十纳秒的差异。
    
	2. 探测器物理响应（漂移时间）

		- **原理**：粒子打入探测器后，产生电荷并被收集需要时间。
		
		- **代码对应**：`MWPC_TS_CORR` (475.0 ns) 
		
		- **解释**：
		
		    - **硅 (DSSD)**：电荷收集极快（几纳秒）。
		    
		    - **气体 (MWPC)**：MWPC 是多丝正比室，气体探测器。粒子电离气体后，电子要在电场中慢慢“漂移”到阳极丝，这个过程非常慢（相对而言）。所以 MWPC 的信号天生比 DSSD 慢几百纳秒，必须减去这 475ns 才能对齐。
    
	3. 电子学逻辑延迟与触发机制
	
		- **原理**：不同的数字化仪（V1724 vs V1730）时钟频率不同（10ns vs 2ns），或者触发逻辑不同（比如有的模块设置为“过阈值即触发”，有的设置为“梯形波成形后触发”）。
		
		- **代码对应**：`GE_TS_CORR` (-160.0 ns)
		
		- Ge 探测器通常使用慢成形参数来保证能量分辨率。这里的负值（-160）意味着 Ge 的触发信号比 DSSD **早**被记录下来（或者被系统“认为”早了）。这通常是不同型号插件板之间的同步相位差导致的。
    
	4. ~~系统状态逻辑（非物理符合，不用管，这个东西本次实验中没有）~~

		1. 为什么会有“束流开 (BeamOn)”和“束流关 (BeamOff)”？
			
			- **实验背景**： 在核反应实验中，当束流打在靶上时（Beam On），会有大量背景噪声（本底）。如果您想观测一个寿命很短的原子核的衰变，最好的办法是：**先打一会束流产生原子核，然后把束流关掉（Beam Off），在安静的环境下测量衰变。**
			
			- **信号延迟的来源**：
			    
			    - **逻辑信号**：加速器控制室会发出一路 TTL 逻辑信号，告诉 DAQ：“我现在开始给束流了”（上升沿）或者“我现在关掉束流了”（下降沿）。这个信号走很长的电缆过来。
			    
			    - **物理延迟**：从逻辑信号发出，到离子源反应、加速、飞行几十米到达探测器，有一个巨大的时间差。
			    
			- **数值含义 (70e3 = 70us)**： 这里的 `70e3` 并不是说束流开关只持续了这么短，而是**“修正偏移量”**。
			
		    - **BEAMOn_TS_CORR (70us)**：表示从记录到“束流开启逻辑信号”开始，还要过 70us，真实的粒子才会稳定打到靶上。
		    
		    - **BEAMOff_TS_CORR (7us)**：表示从记录到“束流关闭逻辑信号”开始，再过 7us，束流就真的切断了。
		    
		2. 为什么会有“靶开 (TargetOn)”和“靶关 (TargetOff)”？

			对应代码中提到的 **Rotating Target（旋转靶）**。

			- **实验背景**： 高强度束流打在靶上会产生巨大的热量，如果靶子不动，瞬间就会被烧穿。所以实验中使用**旋转靶轮**，让束流打在快速旋转的圆盘上散热。
    
			- **“开”与“关”的含义**： 靶轮不是铁板一块，它通常是一个轮辐状结构，或者分成了很多扇区（Target Frames）。
    
		    - **Target On**：束流打在靶材料（如铀靶）上——**有效数据**。
        
		    - **Target Off**：束流打在靶框（金属支架）上，或者两个靶块之间的缝隙——**无效数据/噪声**。
        
			- **为什么延迟这么大 (350e3 = 350us)**？ 这是**机械旋转的时间**。
    
			    - 靶轮上有一个传感器（比如光电门），每转一圈给一个“零点信号”。
        
			    - `350e3` 表示：收到零点信号后，靶轮还需要转 350微秒，第 1 号靶块才会转到束流的位置。
        
			    - `650e3` 表示：再转到 650微秒时，第 1 号靶块就转过去了（束流开始打到靶框了）。
    
3. 怎么修正？

	这是实际操作层面，通常分为“离线刻度”（找参数）和“在线应用”（写代码）两步。

	- 找到修正值（刻度过程，前边找时间窗也是这样干的）

		1. **做符合实验**：使用一个放射源（如 $^{241}\text{Am}$ 或在束流实验中找一个已知的高产额反应通道），确保会有粒子同时打在 DSSD 和待刻度探测器（如 MWPC）上。
    
		2. **画时间差谱**：在 ROOT 中画一个直方图，横坐标是 $T_{MWPC} - T_{DSSD}$。
    
		3. **找峰位**：你会发现这个分布不是以 0 为中心的，而是形成一个高斯峰，峰中心可能在 475ns 的位置。
			
			- 会高斯分布是因为它不一定这么精准，每次都慢个475ns，有时候可能是476，有时候可能是474，总之是在475附近的小范围有浮动
    
		4. **确定参数**：这个 **475** 就是修正系数。

### 波形死时间区域（用坐标窗判断）:
```cpp
//用于判断是否处波形死时间范围内
#define DEADTIME_X_LOW     21.8e3
#define DEADTIME_X_HIGH    26e3
#define DEADTIME_Y_LOW     16.8e3
#define DEADTIME_Y_HIGH    21e3
```

**电子学死时间（Dead Time）恢复窗口** 的参数，定义了当探测器的一面（比如X面）因为上一个大信号饱和而“瞎”了的时候，我们去“抢救”另一面（Y面）看到的真实信号的时间范围。

在重离子注入实验中，经常发生以下物理过程：

1. 注入：一个高能重离子打入 DSSD，释放巨大能量（比如几十 MeV）。这会导致前置放大器产生一个巨大的脉冲，常常导致电子学进入 **饱和** 状态。
    
2. 快速衰变：注入的核素寿命极短（比如几微秒），在注入后极短时间内发生 $\alpha$ 衰变。

**问题在于**：X 面和 Y 面的电子学恢复速度可能不同。

- 假设 X 面电子学恢复慢，注入后 30µs 内都是“死”的（无法处理新信号）。
    
- 假设 Y 面电子学恢复快，注入后 10µs 就活了。
    
- 如果衰变发生在第 20µs：**Y 面看到了信号，但 X 面还是瞎的**。

所以就得到一个“单面事件”（只有 Y，没有 X），通常这种事件因为无法确定位置（Pixel）会被丢弃，但在这里不是。

- **`DEADTIME_X_LOW` (21.8µs) ~ `DEADTIME_X_HIGH` (26µs)**：X 面电子学的“盲区”或特定恢复时间窗。
    
- **`DEADTIME_Y_LOW` (16.8µs) ~ `DEADTIME_Y_HIGH` (21µs)**：Y 面电子学的“盲区”或特定恢复时间窗。

在mapping.cpp中Analyze_Coin_Event函数中被调用
```
if( Data_DSSDX.mul == 0 || Data_DSSDY.mul==0)
{
    Deadtime_Flag = 1;
    if(Check_DeadTime_Event() < 0)
        return;
}
```
如果有一面没有信号，就检查一下是不是处于死时间，如果是就返回死时间事件的真实位置，并把另一面的时间给这个事件，如果没找到就返回-1，代码执行 `return`，`Analyze_Coin_Event` 函数直接结束，不再往下执行。这个事件被彻底**丢弃**了，因为它既只有一面，又和历史事件扯不上关系，被认定为纯噪声。

这两个时间窗在 `mapping.cpp` 的 `Check_DeadTime_Event()` 函数中起到了 **“坐标继承”** 的作用。

当程序发现一个 **单面信号**（比如只有 X，或者只有 Y）时，它会执行以下逻辑：

**如果只有 X 信号 (Y 面丢了)**，程序就去找Y 面是不是因为上一个大信号还在“死时间”里

1. **回溯历史**：程序查看该 X 条带对应的所有像素的历史记录 (`Data_chain_array`)。
    
2. **计算时间差**：`ts_diff` = 当前时间`(ts_now)` - 上一个历史事件时间`(previous_Ts)`
    
3. **判定**：
```
// 检查时间差是否落在 Y 的死时间窗口内
if(ts_diff > DEADTIME_Y_LOW && ts_diff < DEADTIME_Y_HIGH)
{
    // 如果是，说明 Y 面不仅丢了，而且丢得很有规律（符合死时间特征）
    // 强制把上一个事件的 Y 坐标赋给当前事件
    Data_DSSDY.Ch[0] = idx; 
    return idx; // 抢救成功！
}
```

**如果只有 Y 信号 (X 面丢了)** , 就去找 X 面是不是还在饱和恢复中，流程同上

这些数值不是算出来的，而是**通过实验数据测出来的**。

- 查看 `T_current - T_previous` 的时间差谱， 对于单面事件找一个明显的分布峰，集中在 20µs 左右。这代表了前放放电恢复并重新基线稳定所需的时间。
    
- **X 和 Y 不一样**：因为 DSSD 的 X 面（P-side）和 Y 面（N-side）的物理特性（电容不同）以及可能使用的电子学模块增益不同，导致它们的恢复时间不同。
    
- 代码中 `DEADTIME_X` 比 `DEADTIME_Y` 的时间窗稍晚且稍长，说明该实验中 X 面的电子学恢复比 Y 面更慢。

因为它发生在 `mapping.cpp` 的事件分析阶段，所以它会影响：

- **`*_map.root`**：某些本来“只有 X 或只有 Y”的事件，会被补成“X+Y”，写入 map 的 DSSD_XPos/YPos 等信息就会变化。
    
- **`*_chain.root`**：最关键——链是按像素 `pixel_num` 存在 `Data_chain_array[pixel_num]` 里的。补了缺失面才能确定像素，不然链会断、或者事件直接 `return` 掉。
    
- **不影响 `*_tmp.root`**（tmp 在 `SortEvent.cpp` 生成，早于 mapping/chain 逻辑）

```cpp
           Data_MWPC.E[kk] < Implant_MWPC_Range[MWPC_idx][1])
            MWPC_OK = true;
    }
    if(MWPC_OK == false)
        return;

	...
}
```

1. **把“注入事件”和“衰变事件”区分开**

	- **注入（implant）**：需要 `Data_MWPC.mul > 0` 且 MWPC 落在 `Implant_MWPC_Range`，并且 DSSD 能量还得落在 `Implant_Energy_Range`
    
	- **衰变（decay）**：要求 `Data_MWPC.mul == 0` 且 `Data_Veto.mul == 0`（并在时间窗内追加到已有链）
    
	也就是说：`Implant_MWPC_Range` 是 implant 判定的关键门之一。

2. **防止“假 MWPC”导致误把衰变/噪声当注入**

	- 比如 MWPC 有噪声、串扰、基线漂移，可能出现“看起来 MWPC 有 hit”，但幅度不对。  
	
	- 用 `Implant_MWPC_Range` 约束后，这类事件会被 `MWPC_OK == false -> return` 直接丢弃，不会触发“清旧链 + 建新链”的逻辑。

3. **保证建链像素的“注入点”是可靠的**

	- 因为后面会对该 pixel 的历史链做清理、封存（写入 tr_chain）甚至删掉 `chain_mul==1` 的缓存链，然后 push 一个新的 `ptr_data_new` 作为链起点。如果 implant 判错，会导致：
	
		- 错误清掉真正的链
	    
		- 把链起点放错事件
	    
		- 后续 decay 全部关联错
    
	- `Implant_MWPC_Range` 就是在降低这种灾难性误配的概率。

**它影响哪些输出文件

- **`*_chain.root` 影响最大**：决定哪些事件被当作“注入点”去开启/刷新链。
    
- **`*_map.root` 间接影响**：map 里会记录 MWPC 的 mul/E 等；但“注入判定”主要改变的是 chain 逻辑与链缓存清理策略。
    
不影响 `*_tmp.root`（tmp 在 `SortEvent.cpp` 里生成）。

### 波形处理算法相关参数（FT / 梯形 / CFD）
```cpp
//快速滤波(FT)和梯形算法(SL)相关参数
#define FT_RS  10       //in 10ns, 滤波的积分段长度
#define FT_GAP  5      //in 10ns,
#define SL_RS  100       //in 10ns,
#define SL_GAP  50       //in 10ns,
#define CFD_DELAY  5     // CFD的delay
#define CFD_F      0.3

#define DECAY_TIME   40000  //信号的衰变时间，设置成跟实际信号的decay time一致就好
#define NSAMPLE_PRETRIGGER   185
#define NSAMPLE_V1724   3000
#define NSAMPLE_V1730   1600
#define NSAMPLE_MAX      NSAMPLE_V1724
```
- FT 把原始的前放信号（通常是指数衰减的形状）整形成一个 **窄梯形**。
	
	- `FT_RS` (Fast Trigger Rise Time / 上升时间)这是滤波器的**积分窗口长度**，平均多少个点来计算上升沿
        
        - 值越小：梯形越窄，**分辨堆积信号（Pile-up）的能力越强**，时间定位越准，但噪声大。
            
        - 值越大：噪声越小，但梯形变宽，两个靠得近的信号容易混在一起。
	    
	- **`FT_GAP` (Fast Trigger Gap / 平顶时间)**，梯形的**平顶宽度**。
        
	    - 必须设置得比原始信号的物理上升时间（Rise Time）稍长一点点。
        
	    - 原始信号从 0 升到最大值需要时间。如果 GAP 设得太短（小于信号上升时间），梯形还没升到顶就被拉下来了，导致幅度亏损（Ballistic Deficit）。
		
	- **总宽度** = 2 * `FT_RS` + `FT_GAP` = 25个点 (250ns)。这意味着只要两个信号间隔超过 250ns，这个算法就能把它们分开。
		
	- 在哪里用到
		
		- WaveProcess.cpp中的[[离线数据分析程序详解#`Get_Trace_filter`：计算数字滤波输出（fast filter）|Get_Trace_filter]]
		
		- WaveProcess.cpp中的[[离线数据分析程序详解#`Get_Trace_Trigger_RCCR2`：用差分形成 Trigger_Trace，再找零交叉|Get_Trace_Trigger_RCCR2]]
		
		- WaveProcess.cpp中的[[离线数据分析程序详解#`Search_Trigger_Point`：统一的“触发点搜索器”|Search_Trigger_Point]]
		
		- SortEvent.cpp中的[[离线数据分析程序详解#` ifdef DECODE_WAVE_PHA`：对 pileup 波形做“重新触发 + 重算 PHA + 拆分成多条事件”|DECODE_WAVE_PHA]]

- SL 通道使用很长的积分时间来压制噪声，因此生成的梯形波很宽
	
	- 用在哪里
		
		- WaveProcess.cpp中的[[离线数据分析程序详解#`GetPHADLK`：PHA 差分核|GetPHDLK]]
		
		- WaveProcess.cpp中的[[离线数据分析程序详解#`GetPHAResult`：数字 PHA（梯形/CR-RC 类）算法算能量|GetPHAResult]]
		
		- 通俗一点说，GetPHADLK的含义是从原始波形中求出一个**二阶导d_lk**，然后回到GetPHAResult中对二阶导积分**得到一阶导也就是我们想要的方波Pn**，对一阶导再积分**得到PHA_Result**也就是我们想要的最终的梯形波
			
			- <font color=red>这个二阶导不是原始波形的二阶导，是未来要生成的梯形波的二阶导</font>

- **CFD恒比定时甄别器**
	
	- **`CFD_F` (0.3)**：**恒比分数 (Fraction)**。
	    
	    - **含义**：表示触发点设定在信号最大幅度的 **30%** 处。
        
	    - **物理意义**：对于半导体探测器（如 DSSD 或锗），信号上升沿的 30%-40% 区域通常线性度最好，且受噪声影响较小。在这个比例触发，能获得最佳的时间分辨率。
        
	- **`CFD_DELAY` (5)**：**延迟时间 (Delay)**。
	    
	    - **含义**：单位是采样点。如果1个点=10ns，那么延迟就是 **50ns**。
        
	    - **物理意义**：这是构造“过零点”所需的延迟量。它必须比信号的上升时间短，用于将原始单极性信号转换为双极性信号。
		
	- 用在哪里
		
		- WaveProcess.cpp中的[[离线数据分析程序详解#`Get_Trace_Trigger_CFD`：CFD 触发波形 + 找零交叉|Get_Trace_Trigger_CFD]]
		
	- CFD在程序中并未被调用（写好了要用到CFD的函数但没调用）

- decay_time 前置放大器的衰减常数 ($\tau$)

	- 当探测器（如 DSSD 或Ge探测器）探测到粒子时，产生的电荷经过**前置放大器**输出的信号并不是一个完美的方波，而是一个具有快上升沿（Rise time）和**慢指数衰减拖尾**（Decay time）的信号（通常像鲨鱼鳍的形状）。$$V(t) \propto e^{-t / \tau}$$代码中通常以 **采样点数** 为单位（例如，如果采样率是 100MHz，1个点=10ns，如果 $\tau=50\mu s$，则 `DECAY_TIME` 约为 5000）。
    
    - 如果不处理这个长尾巴，当两个信号靠得很近时，第二个信号会骑在第一个信号的尾巴上（基线被抬高），导致第二个信号的能量算错（偏大）。这种现象叫 **堆积 (Pile-up)**。
    
    - 需要通过算法把这个指数尾巴强行“拉平”回零基线，这个过程在信号处理中叫 **极零相消**。
    
    - 在代码中的作用：极零相消 (PZC)
		
		1. 在梯形成形算法中 (核心作用)，用于把指数波变成平顶梯形波，在[[离线数据分析程序详解#`GetPHAResult`：数字 PHA（梯形/CR-RC 类）算法算能量|GetPHAResult]]和[[离线数据分析程序详解#`GetPHADLK`：PHA 差分核|GetPHADLK]]中
			
			1. **初始化**：
			    ```
			    WaveProcess() {
			        Decay_Time = DECAY_TIME; // 将宏定义传给类成员
			        // ...
			    }
			    ```
			    
			2. **极零相消公式**：在`GetPHAResult`函数中：
			    ```
			    // WaveProcess.cpp
			    PHA_Trace[i] = PHA_Trace[i-1] + Pn + d_lk * Decay_Time;
			    ```
			    
			    - `d_lk`：是微分项。
			        
			    - `Pn`：是积分项。
			        
			    - `+ d_lk * Decay_Time`：**这就是极零相消项！**
			        
			    - **原理**：梯形滤波本质上是对信号进行积分。如果你直接对一个指数衰减信号积分，你会得到一个顶部倾斜的梯形。通过加上 `d_lk * Decay_Time` 这个修正项，正好能抵消指数衰减带来的顶部倾斜，使梯形波的顶部变得**水平**。
			    
			3. **能量归一化**： 在 `GetPHADLK` 函数中：
			    ```
			    return ret/(PHA_RS*Decay_Time + PHA_RS);
			    ```
			    
			    这里除以 `Decay_Time` 是为了归一化增益，确保算出来的能量值不会因为 `Decay_Time` 参数的变化而剧烈波动
				
			- **如果 `DECAY_TIME` 设置错误：**
				
				- **设大了**：梯形波的平顶会**上翘** (Overshoot)，能量偏大。
				    
				- **设小了**：梯形波的平顶会**下塌** (Undershoot)，能量偏小。
				    
				- **设准了**：梯形波的平顶是**水平**的，能量分辨率最佳。
		2. 在波形拟合算法中 (DECODE_WAVE_FIT)，在[[离线数据分析程序详解#`fit_Pulse`：单个脉冲形状模型|fit_Pulse]]中
			```
			Double_t fit_Pulse(Double_t *x, Double_t *par) {
			    // ...
			    // 标准的双指数脉冲公式： A * (exp(-t/Decay) - exp(-t/Rise))
			    fun = par[0]*(TMath::Exp(-(x[0]-par[1])/DECAY_TIME) - TMath::Exp(-(x[0]-par[1])/par[2]));
			    return fun;
			};
			```
			
			在这里，`DECAY_TIME` 直接作为拟合函数的**固定参数**。
			
			- 程序假设所有信号的衰减常数都是 `DECAY_TIME`。
			    
			- 拟合程序只需要去算幅度 (`par[0]`) 和 时间 (`par[1]`)，从而减少了自由度，让拟合更稳定、更快。

	- 比如说存储的第一个能量值就是`FIT_RESULT.E[0]`

### `fit_Pulse`：单个脉冲形状模型
```cpp
//------------------------------------------
Double_t fit_Pulse(Double_t *x, Double_t *par) {
	Double_t fun;
	if(x[0] < par[1])
		fun = 0;
	else
		fun = par[0]*(TMath::Exp(-(x[0]-par[1])/DECAY_TIME) - TMath::Exp(-(x[0]-par[1])/par[2]));
	return fun;
};
```

在这里，`DECAY_TIME` 直接作为拟合函数的**固定参数**。

- 程序假设所有信号的衰减常数都是 `DECAY_TIME`。
    
- 拟合程序只需要去算幅度 (`par[0]`) 和 时间 (`par[1]`)，从而减少了自由度，让拟合更稳定、更快

### `Get_Trace_filter`：计算数字滤波输出（fast filter）
```cpp
int WaveProcess::Get_Trace_filter(double *ret_trace)
{
	// 检查波形是否存在
	if(nSample ==0)
	//nSample是这条波形的采样点数，在SetWaveTrace()里设置
	{
		printf("trace is NULL, plx use SetWaveTrace() func first\n");
		return(-1);
	}
	
	double fsum0, fsum1;
	unsigned int offset, ii;

	//检查波形长度是否足够长以支持滤波运算
	if(nSample < ((2*Filter_RS + Filter_GAP)*2))
	{
		printf("nSample is too short\n");
		return(-2);
	}

//	PreFilter(NULL);
    // Compute fast filter response
    offset = 2*Filter_RS + Filter_GAP - 1;//算法里没有用到，先别管了
    fsum0=0;
    
    Filter_Trace[0] = 0;//滤波输出的第一个点设为 0（基准点）
    for(ii = 1; ii < nSample; ++ii)
    {
        fsum0 = GetTracePoint(ii-Filter_RS-Filter_GAP) - GetTracePoint(ii-2*Filter_RS-Filter_GAP);
        fsum1 = GetTracePoint(ii) - GetTracePoint(ii-Filter_RS);
        Filter_Trace[ii] = Filter_Trace[ii-1] + (1.0*fsum1-fsum0)/Filter_RS*10;
        //Filter_Trace[ii] = Filter_Trace[ii-1] + (1.0*fsum1-fsum0);
    }

	//if required ret_trace
	if(ret_trace != NULL)
	{
		memcpy(ret_trace,Filter_Trace,sizeof(Double_t)*nSample);
	}
	Filter_OK=true;
    return nSample;
}
```
**核心循环部分**解释

- [[离线数据分析程序详解#`GetTracePoint`：越界时返回 baseline|GetTracePoint]]

- 举个例子，比如说现在有100个点，前9个都是0，第10个突变到100
	
	- ii<=9时，fsum0和fsum1都=0
	
	- ii=10时，fsum1突变到100，而fsum0还是0，所以就在`Filter_Trace[ii-1]`的基础上加上`(1.0*fsum1-fsum0)/Filter_RS*10`，也就是$+2\times 10$
		- 除以`Filter`就回到原来高度了，保证滤波前后高度不变![[Pasted image 20260126165036.png]]
	
	- 10<=ii<=14这些点，全部都比前一个点的数值大了$2\times 10$，这就是上升沿
	
	- 15<=ii<=19这些点，fsum0和fsum1都=0，全部都和前一个点的数值一样大，这就是平顶部分
	
	- 20<=ii<=24这些点，fsum1仍然是0，但fsum1突变到100，也就是$-2\times 10$，这些点全都比前一个点的数值小了$2\times 10$，这就是下降沿

- **衰减 (`Filter_Trace[x] * CFD_f`)**：将原信号缩小成 30% 的小信号。
    
- **延迟反相 (`- Filter_Trace[x-CFD_delay]`)**：将原信号延迟 50ns，反转（变成负的），还是 100% 幅度。
    
- **合成 (`Trigger_Trace`)**：两者相加。
    
    - 当信号刚开始上升，延迟的大负信号还没来，总和是正的。
        
    - 当延迟的大负信号到来后，总和会迅速变成负的。
        
    - **过零点 (Zero Crossing)**：正负抵消的那个瞬间。数学上可以证明，这个瞬间严格对应原始信号上升沿的 30% 时刻
	
- 跟下边这个`Get_Trace_Trigger_RCCR2`是几乎一样的过程，只不过把延迟量换成了`CFD_delay`
### `Get_Trace_Trigger_RCCR2`：用差分形成 Trigger_Trace，再找零交叉

```cpp
	{

	}
	
    for(ii = Filter_GAP; ii < nSample; ++ii)
        Trigger_Trace[ii] = Filter_Trace[ii]-Filter_Trace[ii-Filter_GAP];
    for(ii = 0; ii < Filter_GAP; ++ii)

	//if required ret_trace

	//	memcpy(ret_trace, Filter_Trace, sizeof(Double_t)*nSample);

    return Fit_Result.num;

```

前边`Get_Trace_filter`可以拿来算能量，但是梯形波的顶是平的，没办法确定时间，所以这部分我们就把单极性的梯形波转为双极性波

这里不再是算跟上一个点的加和，而是求当前点和gap前点的差值，这样就会得到一个零点，这个零点几乎不受信号幅度的影响，无论幅度大小零点都在这里![[c03f3e53c9253f751115f3f648761e7 1.jpg]]

后边那个循环是边界处理，因为数组最开头的部分（ii-Gap）会变成负数，所以这里简单处理为减去基线(ii-0)

最后的if语句是做个判断，如果需要，输出这个双极性波形。

- 调用[[离线数据分析程序详解#`Search_Trigger_Point`：统一的“触发点搜索器”|Search_Triiger_Point]]，**记录时间**，返回找到的信号个数 (`Fit_Result.num`)。如果返回 `0` 或 `1`，说明没堆积；如果返回 `>1`，说明有堆积，需要开启 PHA/Fit 算法。

**解释一下CFD_delay作为延迟量和Filter_GAP作为延迟量的区别**

这两个函数代表了两种**不同**的物理目的，它们虽然都叫“快速触发”，但计算逻辑和想要提取的信息截然不同，通常二选一调用。

1. 方案 A：RCCR2 算法 (使用 `Filter_GAP`)

	- `Trigger_Trace[ii] = Filter_Trace[ii] - Filter_Trace[ii - Filter_GAP];`
	
	- **使用的延迟量**：`Filter_GAP` (平顶时间，通常较长，如 500ns)。
    
	- **物理原理**：**梯形波微分**。
    
    - 它把一个“梯形”变成了一个“双极性脉冲”。
        
    - 因为延迟量正好是梯形的平顶宽度，所以相减后，**过零点 (Zero Crossing) 出现在梯形平顶的几何中心位置**。
        
	- **目的**：**堆积判别 (Pile-up Rejection)**。
    
    - 它的核心任务是把信号“分得开”。
        
    - 因为它利用了整个平顶的信息，它生成的双极性波形比较宽，不容易受到高频噪声的干扰，**非常稳健**。
        
    - 它主要告诉程序：“这里有一个信号，大概在这个位置”。
	
2. 方案 B：CFD 算法 (使用 `CFD_delay`)

	- `Trigger_Trace[x] = Filter_Trace[x] * CFD_f - Filter_Trace[x - CFD_delay]`
	
	- **使用的延迟量**：`CFD_delay` (CFD 延迟，通常很短，如 50ns)。
    
	- **物理原理**：**恒比定时**。
    
    - 它通过衰减和短延迟相减，强制**过零点出现在信号上升沿的 30% 处**。
        
    - 注意：`CFD_delay` 必须**小于**信号的上升时间 (`Filter_RS`)，否则这套物理机制就失效了。
        
	- **目的**：**高精度定时 (High Precision Timing)**。
    
    - 它的核心任务是“定得准”。
        
    - 它消除了幅度游动（Amplitude Walk），不管信号大还是小，触发时间都死死锁定在波形起始阶段的同一点。
        
    - 对于需要做飞行时间 (TOF) 或符合测量 (Coincidence) 的实验，必须用这个。
    
3. 为什么要有两套？

	- **RCCR2 (`Filter_GAP`)**：
    
    - 延迟大，过零点在 **山顶中央**。
        
    - **优点**：哪怕信号有点噪声，山顶也很宽，不容易丢信号。适合数个数、算能量。
        
    - **缺点**：山顶是平的，确定“正中心”的那个纳秒可能有 10-20ns 的抖动。定时不准。
        
	- **CFD (`CFD_delay`)**：
    
    - 延迟小，过零点在 **山脚半山腰** (上升沿)。
        
    - **优点**：上升沿非常陡峭，过零点极其锋利，时间精度可以达到 <1ns。
        
    - **缺点**：如果噪声太大，容易在平坦的基线处产生误触发。
    
4. 程序内是怎么处理的？

	- 在 `WaveProcess.cpp` 中，`Trigger_Trace` 是同一个数组（共享内存空间）。
	
	- **覆盖逻辑**：
    
    如果程序先调用了 `Get_Trace_Trigger_RCCR2`，数组里装的就是方案 A 的数据。
    
    如果程序紧接着又调用 `Get_Trace_Trigger_CFD`，数组里的数据就被方案 B **覆盖**了。
    
	- **现状分析**：
    
    在目前的 `SortEvent.cpp` 中，**只调用了** `Get_Trace_Trigger_RCCR2`。
    
    这意味着：
    
    1. 程序使用了 **`Filter_GAP`** 作为延迟。
        
    2. 程序目前追求的是**稳健的触发和堆积判别**，而不是极致的时间精度。
        
    3. `CFD_delay` 虽然在 `setup.h` 里定义了，但在当前运行逻辑中，它对应的计算公式**没有被执行**。

### `Search_Trigger_Point`：统一的“触发点搜索器”
```cpp
void WaveProcess::Search_Trigger_Point()
{
	bool thresh_flag=false;
	// 状态标记：告诉程序当前是否处于“信号内部”。 
	// false = 正在寻找信号起点；true = 已经找到起点，正在等待过零点。
	
	Fit_Result.num=0;   // 结果清零：当前找到的信号个数
	Fit_Result.Ts[0]=0; // 结果清零：第一个信号的时间
	int pos_threshold=0;// 记录信号“刚超过阈值”的那个时刻（用来判断是否超时）
	
	// 搜索窗口（超时时间）
	// 如果信号都超过搜索窗口了还没过零，说明这可能是噪声或者堆积太乱了，放弃。
	int search_window=1.2*(2*Filter_RS+Filter_GAP);
	map_trigger.clear();
	for(unsigned int ii = 1; ii < nSample; ++ii)
	{
		// 【情况一：发现信号起搏】 
		// 还没进入信号状态 (thresh_flag==false) 并且 当前点超过了触发阈值
		if(thresh_flag == false && Trigger_Trace[ii] > Filter_Threshold)
		{
			thresh_flag = true;
			pos_threshold = ii;
		}
		
		// 【情况二：超时重置】 
		// 已经在信号里了 (true)，但是时间过了太久 (超过 search_window) 还没过零。
		else if(thresh_flag == true && (ii - pos_threshold) > 2.0*search_window)
			thresh_flag = false;// 算了，这不是个好信号，重置状态，找下一个。
		
		// 【情况三：捕捉过零点 (Trigger Point)】 
		// 已经在信号里 (true) 并且 发生了过零 (当前点和前一个点符号相反，乘积 <= 0)
		else if(thresh_flag == true && Trigger_Trace[ii] * Trigger_Trace[ii-1] <= 0 )
		{
			// 边界检查：防止越界（不要太靠前，也不要太靠后）
			if( ii > (Filter_GAP+Filter_RS+20) && ii < nSample-10 )
			{
		        // *** 核心存储逻辑 *** 
		        // map_trigger 是一个 multimap<double, int> (Key=幅度, Value=时间) 
		        
		        // Key (幅度): Trigger_Trace[ii-(Filter_GAP+Filter_RS)/2] 
		        // 解释：去过零点前面一点的地方取一个值作为“幅度代表”，用于后续按能量排序。 
		        
		        // Value (时间): ii - Filter_GAP - Filter_RS 
		        // 解释：这是我们在上一个问题里讨论的【时间倒推】。 
		        // ii 是过零时刻，减去成形参数，算出物理上的 T0 时刻。
		        map_trigger.emplace(Trigger_Trace[ii-(Filter_GAP+Filter_RS)/2], ii - Filter_GAP - Filter_RS);
			}
			thresh_flag = false;// 找到了一个完整信号，闭环。重置状态，准备找下一个堆积信号。
		}
	}

	//sort by amplitude & ts
	// 如果找到了至少一个信号
	if(map_trigger.size() > 0)
	{
		Fit_Result.num=0;
		
		// map_trigger 默认是按 Key (幅度) 从小到大排序的。
		auto iter = map_trigger.end();// 指向最后（也就是幅度最大）的后面
		map_Fit.clear();// 这个map的Key是时间！！！
		
		// 【步骤 A：挑出幅度最大的前 N 个】
		do{
			iter--;// 往前挪，拿到幅度最大的那个
			
			// 存入 map_Fit 
			// 注意：这里把 iter->second (时间) 变成了 map_Fit 的 Key。 
			// map 容器会自动根据 Key (时间) 进行从小到大的排序。
			map_Fit[iter->second] = iter->first;
			
			Fit_Result.num++;// 计数
		}while(Fit_Result.num<MAX_NUM_PILEUP && iter != map_trigger.begin());// 循环条件：还没找够MAX_NUM_PILEUP且容器里还有数据

// 到这里，map_Fit 里装着幅度最大的几个信号，并且已经自动按时间排好序了。

		//sort by time
		// 【步骤 B：填入最终结果数组】
		Fit_Result.num=0;// 重新计数，这次是往数组里填
		auto iter_ts = map_Fit.begin();// 从最早的时间开始遍历
		while(Fit_Result.num < MAX_NUM_PILEUP && iter_ts != map_Fit.end())
		{
			// iter_ts->first 是时间 (因为 map_Fit 的 Key 是时间)
			Fit_Result.Ts[Fit_Result.num] = iter_ts->first;
			Fit_Result.num++;// 有效信号数 +1
			iter_ts++;
		}
	}
}
```

#### 第一阶段：风平浪静（寻找信号起点）

**场景**：`ii = 1` 到 `ii = 20`。波形值都在 0 左右，远小于阈值 100。

- **代码逻辑**：
    
    - `if(thresh_flag == false && Trigger_Trace[ii] > 100)` -> **不满足**（因为波形值 < 100）。
        
    - **结果**：什么都不做，继续下一个点。`thresh_flag` 保持 `false`。
        

#### 第二阶段：发现信号（触发阈值）

**场景**：`ii = 21`。波形值突然跳到了 **150**（大于 100）。

- **代码逻辑**：
    
    - **`if(thresh_flag == false && Trigger_Trace[ii] > 100)`**
        
        - `thresh_flag` 是 `false`？**是**。
            
        - 当前值 150 > 100？**是**。
            
        - **满足条件！**
            
- **执行动作**：
    
    - `thresh_flag = true;` (标记：**注意！我们进入信号内部了！**开始准备抓过零点。)
        
    - `pos_threshold = 21;` (记下：是在第 21 个点进来的。)
        
#### 第三阶段：等待过零（信号持续中）

**场景**：`ii = 22` 到 `ii = 29`。波形值从 150 -> 300 -> 500 -> 200 -> 50。虽然在下降，但还是**正值**。

- **代码逻辑**：
    
    - `if(thresh_flag == false ...)` -> **不满足**（因为 `flag` 已经是 `true` 了）。
        
    - `else if(... > 2.0*search_window)` -> **不满足**（还没超时）。
        
    - **`else if(thresh_flag == true && Trigger_Trace[ii] * Trigger_Trace[ii-1] <= 0)`**
        
        - `flag` 是 `true`？**是**。
            
        - 当前点(正) * 前一点(正) <= 0？**不是**（正数乘正数还是正数，没有过零）。
            
        - **不满足条件**。
            
- **结果**：继续循环，耐心等待。
    
#### 第四阶段：抓捕时刻（过零点出现！）

**场景**：`ii = 30`。波形值变成了 **-50**（负值）。 注意：`ii=29` 时是 **+50**。从 +50 变到 -50，这就是**过零**。

- **代码逻辑**：
    
    - 前两个 `if` 都不满足。
        
    - **`else if(thresh_flag == true && Trigger_Trace[ii] * Trigger_Trace[ii-1] <= 0)`**
        
        - `flag` 是 `true`？**是**。
            
        - `Trigger_Trace[30]` (-50) * `Trigger_Trace[29]` (+50) = -2500。
            
        - -2500 <= 0？**满足！**（发现正负反转了）。
            
- **执行动作**：
    
    1. **记录**：把这个信号存下来！
        
        - 幅度取值：取前面某个点（比如峰值附近）的幅度作为 Key。
            
        - 时间计算：`T0 = 30 - GAP - RS`。
            
    2. **重置**：
        
        - `thresh_flag = false;` (标记：**这个信号处理完了**，恢复待机状态，准备抓下一个信号。)
            
#### 第五阶段：防止超时（异常情况）

**场景假设**：如果波形在第 21 点超过阈值后，一直保持正值（比如基线飘起来了），过了几百个点都没掉到 0 以下。

- **代码逻辑**：
    
    - 当 `ii` 增加到很大（比如 `ii = 100`），`ii - pos_threshold` (100 - 21 = 79) 超过了设定的 `search_window` (假设设的是 50)。
        
    - **`else if(thresh_flag == true && (ii - pos_threshold) > 2.0*search_window)`**
        
        - **满足条件！**
            
- **执行动作**：
    
    - `thresh_flag = false;` (强制重置：**等太久了，这肯定不是个好信号，放弃它，重新开始。**)

这个函数会一直循环直到找齐MAX_NUM_PILEUP=10个信号，由于FILTER_THRESHOLD设置的比较小，有可能会把一些较高噪声给包含进来，但是不用担心，在SortEvent.cpp中[[离线数据分析程序详解#` ifdef FILE_TIME_SORT`：进一步“清洗 + 时间修正 + 刻度 + 可选 pileup 重算”，然后写入 shm|FILE_TIME_SORT]]会再进行一次判断，把低于DSSD能量阈值的信号给剔除出去

理想情况下得到平顶的梯形波，但实际上得到的是指数衰减的波

平顶意味着**积分波形的斜率（导数）为 0**。
$$\frac{d(\text{Output})}{dt} = 0$$
**为什么直接积分会是弯的，积分斜率慢慢变到0**![[3fd3aa4e53ee3876e07fe4cbe2a0a82 1.jpg]]

假设探测器过来的信号 $V(t)$ 是一个指数衰减信号：$V(t) = A \cdot e^{-t/\tau}$，其中 $\tau$ 就是 `Decay_Time`。

如果我们只做积分$\text{Output}_{\text{int}}(t) = \int V(t) dt$

这个积分结果的斜率是$\frac{d}{dt} (\text{Output}_{\text{int}}) = V(t) = A \cdot e^{-t/\tau}$

**问题出现了**：积分结果的斜率在不断变小，不是嘎一下归零的。

**那怎么让斜率归零？**

为了让总输出的斜率为 0（变平），我们需要引入一个 **修正项 (Correction)**，让它的斜率正好能抵消掉积分项的斜率。

我们构建总输出公式：$$Y(t) = \underbrace{\int V(t) dt}_{\text{积分项}} + \underbrace{X(t)}_{\text{修正项}}$$我们希望 $Y'(t) = 0$，也就是：
$$Y'(t) = V(t) + X'(t) = 0$$
$$\Rightarrow X'(t) = -V(t)$$
我们需要找一个修正项 $X(t)$，它的导数等于负的输入信号。

对于指数信号 $V(t) = A e^{-t/\tau}$，它的导数是什么？
$$V'(t) = -\frac{1}{\tau} \cdot \underbrace{A e^{-t/\tau}}_{V(t)} = -\frac{1}{\tau} V(t)$$
移项一下：$V(t) = -\tau \cdot V'(t)$

所以就得到结论：**信号本身 ($V$) 和它自己的导数 ($V'$) 成正比**

如果我们把修正项设为：$X(t) = \tau \cdot V(t)$

也就是 `Decay_Time * Input_Signal`

代码里写的是`PHA_Trace[i] = PHA_Trace[i-1] + Pn + d_lk*Decay_Time`

先明确，信号本身($V$) =对信号积分($Y$)的斜率 

原式可变为$PHA_{Trace}[i]-PHA_{Trace}[i-1]=Pn + d_{lk}\cdot Decay_{Time}$

也就是$$d Y_{cur}=d(\underbrace{\int V(t) dt}_{\text{积分项}} + \underbrace{\tau \cdot V(t)}_{\text{修正项}})=V(t)+\tau\cdot dV(t)$$所以 $Pn$ 是恢复后的方波本身也是方波积分的导数，$d_{lk}$ 是Pn上两点高度差

- **`PHA_RS + PHA_Gap`**: 这是梯形波**理论上的平顶结束点**（也就是悬崖边）。
    
    - 从起点出发，走完上升时间 (`RS`)，再走完平顶时间 (`Gap`)，就到了要掉下去的地方。
    
- **`- ave_num`**: **给“平均窗口”留出位置**。
    
    - 不是只采一个点，而是要采 `ave_num`（比如 4个）点求平均。
        
    - 如果从悬崖边开始采，后面 3 个点就掉到悬崖下面去了。
        
    - 所以必须**倒退** `ave_num` 步，确保采的这 4 个点都在平顶上。
    
- **`- 1`**: **安全余量 (Safety Margin)**。
    
    - 因为数字信号处理有**时间抖动 (Jitter)**。快通道算出来的触发时间 $T_s$ 可能有 $\pm 1$ 个点的误差。
        
    - 这 `-1` 就是**往回缩一步**，宁可采在平顶靠前一点（反正也是平的），也绝对不能一只脚踏空掉下去。

这时候就有聪明的朋友要问了，前边`Filter_Trace`不就是找能量用的吗，为什么还要用PHA再找一次？
	
 因为`Filter_Trace`的上升时间`FT_RS`很短，它的梯形很窄，反应很快，能把两个挨得很近的信号分开（堆积判断能力强），时间定位准，但是积分时间短，包含的噪声多，能量分辨很差
	
 而`PHA_Trace`的上升时间`SL_RS`通常很长，梯形很宽，噪声被平均掉了，所以能量分辨率极高
	
 因此把`Filter_Trace`当作找时间的预处理，在`Trigger_Trace`找时间，在`PHA_Trace`找能量

### `GetPHADLK`：延迟获取Pn的变化率
```cpp
Double_t WaveProcess::GetPHADLK(int idx)
{
	Double_t ret =  0.0;
	int num;
	if( idx < nSample && idx >= 0)
	{
		ret += Input_Trace[idx];//上升项（正贡献）
		num = idx - 1.0*PHA_RS;
		ret -= (num > 0)? Input_Trace[num]: BS_data;
		//第一次延迟项
		num = idx - 1.0*PHA_RS - PHA_Gap;
		ret -= (num > 0)? Input_Trace[num]: BS_data;
		//第二次延迟项
		num = idx - 2.0*PHA_RS - PHA_Gap;
		ret += (num > 0)? Input_Trace[num]: BS_data;
		//第三次延迟项
	}
	return ret/(PHA_RS*Decay_Time + PHA_RS)//归一化
}
```

简化模型，假设输入信号 $V(t)$ 是一个完美的**阶跃信号**（0时刻突然从0变成10），不考虑衰减。

假设参数：

- **$k$ (上升时间)** = 2 (点)
    
- **$l$ (平顶结束)** = 5 (点)

公式：$d_{lk}(t) = V(t) - V(t-2) - V(t-5) + V(t-7)$

**`GetPHADLK` 输出 ($d_{lk}$)**

- $t=0$: $10 - 0 - 0 + 0 = \mathbf{+10}$ 
    
- $t=1$: $10 - 0 - 0 + 0 = \mathbf{+10}$
    
- $t=2$: $10 - 10 - 0 + 0 = \mathbf{0}$ 
    
- ...中间全是0
    
- $t=5$: $10 - 10 - 10 + 0 = \mathbf{-10}$ 
    
- $t=7$: $10 - 10 - 10 + 10 = \mathbf{0}$ 

**序列状态**：`+10, +10, 0, 0, 0, -10, -10, 0...`

**`Pn` 的计算 (一次积分)**

这就相当于把“加速度”积分变成“速度”。

- $t=0$: $Pn = 0 + 10 = \mathbf{10}$
    
- $t=1$: $Pn = 10 + 10 = \mathbf{20}$ 
    
- $t=2$: $Pn = 20 + 0 = \mathbf{20}$ (速度保持)
    
- $t=3$: $Pn = 20 + 0 = \mathbf{20}$
    
- $t=4$: $Pn = 20 + 0 = \mathbf{20}$
    
- $t=5$: $Pn = 20 - 10 = \mathbf{10}$
    
- $t=6$: $Pn = 10 - 10 = \mathbf{0}$

**序列状态**：`10, 20, 20, 20, 20, 10, 0...`**现在就变成一个方波了**

`return ret/(PHA_RS*Decay_Time + PHA_RS)`归一化
$$\text{分母} = \text{PHA\_RS} \times (\text{Decay\_Time} + 1)$$
这个归一化因子是由两个独立的物理过程产生的增益组成的。为了让算出来的能量值跟原始信号的幅度保持一致，需要把这两个增益除掉。

1. `PHA_RS`（积分增益），在快速滤波那块讲过一次了，有图，自己去找

2. `Decay_Time + 1`（极零相消增益）
    
    ```
    // 简化的逻辑概念
    Total_Slope = Pn + d_lk * Decay_Time;
    ```
    
    这里其实包含了两部分贡献：
    
    1. **`Pn` (原始信号贡献)**：对应系数 **1**。
        
    2. **`d_lk * Decay_Time` (修正项贡献)**：对应系数 **`Decay_Time`**。
    
    当一个真实的信号进来时，算法实际上是把“原本的信号”和“算出来的补偿信号”叠加在了一起。
    
    - 原始信号贡献了 1 份幅度。
        
    - 为了抵消衰减，我们人为补加了 `Decay_Time` 份幅度（通常 `Decay_Time` 是几千，这个数很大）。

### `Cal_complexFFT`：手写 Cooley–Tukey FFT
```cpp
int WaveProcess::Cal_complexFFT ( double *data,  int length )
{
	double tempR;
	double tempI;
	double theta;
	double wr;
	double wi;
	double wpr;
	double wpi;
	double wtemp;
	
	unsigned int nComplex;
	unsigned int m;
	unsigned int mmax;
	unsigned int iStep;
	unsigned int i, j;
	
	nComplex = 2 * length;
	
	// Do the bit reversal re-ordering
	j = 0;
	for (i = 0; i < nComplex; i += 2) 
	{
		if (j > i) 
		{
			tempR = data[j];
			tempI = data[j+1];
			data[j]   = data[i];
			data[j+1] = data[i+1];
			data[i]   = tempR;
			data[i+1] = tempI;
		}
		
		m = nComplex / 2;
		while ((m >= 2) && (j >= m)) 
		{
			j = j - m;
			m = m / 2;
		}
		
		j = j + m;
	}
	
	// Do the recursive FFT
	mmax = 2;
	double PI2=3.1415926*2;
	while (nComplex > mmax)
	{
		iStep = 2 * mmax;
		theta = PI2 / mmax;
		wpr = sin(0.5 * theta);
		wpr = -2. * wpr * wpr;
		wpi = sin(theta);
		wr = 1.;
		wi = 0.;
		for (m = 0; m < mmax; m += 2)
		{
			for (i = m; i < nComplex; i += iStep)
			{
				j = i + mmax;
				tempR = wr * data[j]   - wi * data[j+1];
				tempI = wr * data[j+1] + wi * data[j];
				data[j]   = data[i]   - tempR;
				data[j+1] = data[i+1] - tempI;
				data[i]   = data[i]   + tempR;
				data[i+1] = data[i+1] + tempI;
			}
			wtemp = wr;
			wr = wr * wpr - wi * wpi + wr;
			wi = wi * wpr + wtemp * wpi + wi;
		}
		mmax = iStep;
	}
	
	return 0;
}
```

```cpp
inline Double_t Get_Calib_Energy(UInt_t ADC_values,  Int_t ch, UShort_t PU)
{
	if(ch < TOTAL_SI_CH && ADC_values == 0x7FFF)
	{
		if (ch < DSSD_YH_CH_LOW)
			return E_SATURATION_DSSDY;
		else if (ch < SSD_BACK_CH_LOW)
			return E_SATURATION_DSSDYH;
	}
	/// DSSD inter source corr
	Double_t ret = 0.0;
	if(ch < SSD_CH_LOW)
		ret = 1.0*IntSour_Corr_k*Cal_Par[ch][0]+IntSour_Corr_b+1.0*IntSour_Corr_k*Cal_Par[ch][1]*ADC_values;
	//SSD
	else if( ch < SSD_BACK_CH_LOW)
	{
		ret=1.0*Cal_Par[ch][0] + 1.0*Cal_Par[ch][1]*ADC_values;
	}
	else
		ret = 1.0*Cal_Par[ch][0]+1.0*Cal_Par[ch][1]*ADC_values;
	
	if(ret < 0)
		ret = 0;
	return ret;
}
```

#### 在“水位线”之前搜索符合事件：COIN_WINDOW 组包
```cpp
		/// search coin event
		if((Ts_Curr_Max -Ts_Process_Before) >= Search_Map_Windows || finish_flag)
		{
			iter_map = map_DGTZ_data.begin();
			Ts_Process_Before = Ts_Curr_Max - Search_Map_Reject;
			coin_flag = false;
			if(Ts_Coin == 0)
				Ts_Coin = iter_map->first;
			while((iter_map != map_DGTZ_data.end()) && ((iter_map->first < Ts_Process_Before) || finish_flag))
			{
				Ts_Curr = iter_map->first;
				
				if(llabs(Ts_Curr-Ts_Coin) < COIN_WINDOW)  // fill into event
				{
					if(coin_flag == false && iter_map->second.Ch < DSSD_YH_CH_HIGH)
					{
						Ts_Coin = iter_map->first;
						coin_flag = true;
					}
					Get_Particle_Data(&(iter_map->second));
					++iter_map;
				}
				else
				{
#ifdef REANA_COIN_EVENT
					iter_tmp = map_DGTZ_data.lower_bound(iter_map->first);
					iter_tmp--;
					if(iter_map->first <= iter_tmp->first)
					{
						printf("map test error\n");
						exit(0);
					}
					if(iter_map != map_DGTZ_data.end() && iter_tmp != map_DGTZ_data.begin() && iter_map->first - iter_tmp->first < COIN_WINDOW )
					{
						ULong64_t ts_tmp = iter_tmp->first;
						while(iter_tmp != map_DGTZ_data.begin())
						{
							if((iter_map->first - iter_tmp->first >  iter_tmp->first - Ts_Coin) || iter_map->first - iter_tmp->first >= COIN_WINDOW)
							{
								if(iter_tmp->first != ts_tmp)
								{
									Init_Particle_Data();
									iter_begin = map_DGTZ_data.lower_bound(Ts_Coin);
									while(iter_begin != map_DGTZ_data.upper_bound(iter_tmp->first))
									{
										Get_Particle_Data(&(iter_begin->second));
										++iter_begin;
									}
									iter_map = map_DGTZ_data.upper_bound(iter_tmp->first);

								break;
							}

						}
					}

				/// 事件包结束时：对包内各探测器做整理 → 写 map → 做链分析
				if(Data_MWPC.mul > 1)

				tr_map->Fill();
```

这段代码是为了处理 **浮点数坐标**（因为经过 Addback 加和处理后，位置可能是一个小数）。

- **`floor(x)` (地板/向下取整)**：返回不大于 $x$ 的最大整数。
    
- **`ceil(x)` (天花板/向上取整)**： 返回不小于 $x$ 的最小整数。

`pos` 和 `pos_end` 的区别

`Data_DSSDX.Ch[0]` 是一个 `double` 类型的小数。它来源于 `Process_DSSD_MulSignal` 里的 Addback 计算：$$\text{Position} = \text{Ch}_i + \frac{E_{i+1}}{E_i + E_{i+1}}$$这意味着位置可能算出 **10.3** 这种数，代表粒子打在第 10 条和第 11 条之间，偏向第 10 条。

1. 单条事件：粒子正正地打在第 10 条中间，没有发生电荷分际，此时`pos == pos_end`。
    
	- **循环行为**：`for(; pos <= pos_end; ...)` **只执行一次**（只查第 10 条的历史）。这是完全正确的，因为粒子就在第 10 条。
    
2. 跨条事件：粒子打在第 10 条和第 11 条中间的缝隙，电荷被两边分了。经过 Addback 算法后，算出来的重心位置是 **10.4**，此时`pos != pos_end`，两者相差 1。
    
	- **循环行为**：`for(; pos <= pos_end; ...)` **会执行两次**。
    
	    - 第一次查第 10 条的历史。
        
	    - 第二次查第 11 条的历史。
    
- **`pos` 和 `pos_end`**
    
    - 如果相等，说明是单条信号，只查那一根条。
        
    - 如果不等，说明是跨条信号，程序会**同时检查**它左右两边的两根条，看哪一边有死时间内的历史记录，以此来恢复另一面的信号。

```cpp
		for(int ii = 0; ii < ptr_chain_data->MWPC_mul; ++ii)
		{
			ptr_chain_data->MWPC_Ch[ii] = Data_MWPC.Ch[ii];
			ptr_chain_data->MWPC_Ts[ii] = Data_MWPC.Ts[ii];
			ptr_chain_data->MWPC_E[ii] = Data_MWPC.E[ii];
		}
	}
	else
	{
		if(ptr_chain_data->DSSD_Ts[num - 1] == 0)
		{
			ptr_chain_data->Delta_Ts[num] = 0;
			ptr_chain_data->SysTime = ptr_data->SysTime;
		}
		else
			ptr_chain_data->Delta_Ts[num] = ptr_data->Ts[idx] - ptr_chain_data->DSSD_Ts[num-1];
	}
	ptr_chain_data->DSSD_BdNo[num]   = ptr_data->BdNo[idx];
	ptr_chain_data->DSSD_BdEntryNo[num] = ptr_data->BdEntryNo[idx];
	/// SSD and Ge signal
	if(Data_SSD.mul >0 )
	{
		//Sort_By_Ts(&Data_SSD);
		for(int jj=0; jj < Data_SSD.mul && jj + num < MAX_DECAY_CHAIN_NUM ; ++jj)
		{
			ptr_chain_data->SSD_E[num]   = Data_SSD.E[jj];
			ptr_chain_data->SSD_Ts[num]  = Data_SSD.Ts[jj];
			ptr_chain_data->SSD_Pos[num] = Data_SSD.Ch[jj];
			ptr_chain_data->SSD_BdNo[num] = Data_SSD.BdNo[jj];
			ptr_chain_data->SSD_BdEntryNo[num] = Data_SSD.BdEntryNo[jj];
		}
	}
	else if(num > 0  && ptr_chain_data->Delta_Ts[num] < COIN_WINDOW && ptr_chain_data->SSD_Ts[num-1] > 0)
	{
		if(llabs(ptr_chain_data->SSD_Ts[num - 1] - ptr_chain_data->DSSD_Ts[num]) < llabs(ptr_chain_data->SSD_Ts[num - 1] - ptr_chain_data->DSSD_Ts[num -1]))
		{
			ptr_chain_data->SSD_E[num]   = ptr_chain_data->SSD_E[num-1]; 
			ptr_chain_data->SSD_Ts[num]  = ptr_chain_data->SSD_Ts[num-1];
			ptr_chain_data->SSD_Pos[num] = ptr_chain_data->SSD_Pos[num-1];
			ptr_chain_data->SSD_BdNo[num] = ptr_chain_data->SSD_BdNo[num-1];
			ptr_chain_data->SSD_BdEntryNo[num] = ptr_chain_data->SSD_BdEntryNo[num-1];

			ptr_chain_data->SSD_E[num-1]   = 0; 
			ptr_chain_data->SSD_Ts[num-1]  = 0;
			ptr_chain_data->SSD_Pos[num-1] = 0;
			ptr_chain_data->SSD_BdNo[num-1] = 0;
			ptr_chain_data->SSD_BdEntryNo[num-1] = 0;
		}
	}
	// for Ge
	for(int ii=0; ii<Data_Ge.mul && ii<MAX_GE_NUM; ++ii)
	{
		ptr_chain_data->Ge_mul[num]++;
		ptr_chain_data->Ge_E[MAX_GE_NUM*num + ii] = Data_Ge.E[ii];
		ptr_chain_data->Ge_Ch[MAX_GE_NUM*num + ii] = Data_Ge.Ch[ii];
		ptr_chain_data->Ge_Ts[MAX_GE_NUM*num + ii] = Data_Ge.Ts[ii];
	}

	//// target pos and beam status
	ptr_chain_data->Target_Ts[num] =  Data_Beam.Ts[0];
	ptr_chain_data->Target_Num[num] = Data_Beam.PU[0];
	ptr_chain_data->Beam_Status[num] = Data_Beam.Ch[0];
	ptr_chain_data->BeamOn_Ts[num] = Data_Beam.Ts[1];

	ptr_chain_data->chain_mul++;
}
```
